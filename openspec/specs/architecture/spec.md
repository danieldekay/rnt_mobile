## Purpose

## Requirements

### Requirement: Modular Layered Architecture
The system SHALL follow a modular architecture with separation of concerns between API, state management, components, and routing layers.

#### Scenario: API layer isolation
- **GIVEN** the application needs to fetch event data
- **WHEN** components require event data
- **THEN** they access it through the event store, not by calling API functions directly
- **AND** the API layer (`src/lib/api/tribe.ts`) handles all external communication and data normalization
- **AND** the store layer (`src/lib/stores/events.svelte.ts`) manages state and filtering logic

#### Scenario: State management separation
- **GIVEN** the application has UI state that needs to be shared
- **WHEN** multiple components need access to the same data
- **THEN** they subscribe to Svelte stores rather than passing props through deeply nested component trees
- **AND** the event store centralizes event data, loading states, error states, and filter logic
- **AND** components remain focused on presentation and user interaction

#### Scenario: Component reusability
- **GIVEN** the application has common UI patterns
- **WHEN** building new features or modifying existing ones
- **THEN** reusable components from `src/lib/components/` are utilized
- **AND** components are designed to be configurable via props
- **AND** components encapsulate their own styling and behavior
- **AND** changes to a component propagate to all usages

#### Scenario: Route-based organization
- **GIVEN** the application has multiple views
- **WHEN** organizing page-level components
- **THEN** they are placed in the `src/routes/` directory following SvelteKit conventions
- **AND** each route corresponds to a URL path
- **AND** route components handle data loading and page-specific logic
- **AND** shared layout is managed via `+layout.svelte` and `+layout.ts`

### Requirement: Data Flow Direction
The system SHALL maintain a clear, unidirectional data flow from external APIs to UI presentation.

#### Scenario: Data fetching flow
- **GIVEN** the application starts or refreshes data
- **WHEN** the event store initiates a data fetch
- **THEN** the flow is: API functions → normalization → store state → UI components
- **AND** data moves in one direction: external → internal storage → presentation
- **AND** components never call API functions directly
- **AND** components never modify store state directly (only through store actions)

#### Scenario: State update flow
- **GIVEN** the user interacts with filters or search
- **WHEN** filter or search state changes
- **THEN** the flow is: component action → store update → derived data recalculation → component re-render
- **AND** state mutations are centralized in the event store
- **AND** derived data (`$eventStore.events`) is computed reactively
- **AND** components automatically update when their subscriptions change

#### Scenario: Error handling flow
- **GIVEN** an API request fails
- **WHEN** the fetch encounters an error
- **THEN** the flow is: API function → error caught → store error state set → error UI displayed
- **AND** error states are managed in the event store
- **AND** error UI is conditionally rendered based on store state
- **AND** retry flows are initiated through store actions

### Requirement: PWA-First Static Architecture

#### Scenario: Static site generation
- **GIVEN** the application is built for production
- **WHEN** running `npm run build`
- **THEN** a static site is generated in the `./build` directory
- **AND** the build uses `@sveltejs/adapter-static` with prerendering enabled
- **AND** all pages are prerendered at build time where possible
- **AND** dynamic content is handled via client-side fetching

#### Scenario: Service worker integration
- **GIVEN** the application is served over HTTPS
- **WHEN** the PWA loads in a supported browser
- **THEN** a service worker is registered for offline caching and asset management
- **AND** the service worker is configured through the adapter-static configuration
- **AND** updates are handled through Cloudflare Pages' automatic deployment system

#### Scenario: Mobile-first responsive design

- **WHEN** rendering UI components
- **THEN** layouts are designed mobile-first using Tailwind CSS responsive prefixes
- **AND** components adapt to different screen sizes using flex and grid utilities

### Requirement: Technology Stack Consistency
The system SHALL maintain consistency in its technology stack and follow established patterns.

- **GIVEN** the application uses Svelte 5

- **AND** legacy Svelte 4 syntax is avoided in new code

#### Scenario: TypeScript strictness
- **GIVEN** the application uses TypeScript

- **THEN** strict type checking is enforced
- **AND** interfaces are defined for all complex data structures

#### Scenario: Tailwind CSS utility-first styling
- **GIVEN** the application uses Tailwind CSS
- **WHEN** styling components or adding new UI elements
- **THEN** utility classes are used for styling rather than custom CSS where possible
- **AND** custom CSS is reserved for complex animations or component-specific styling
- **AND** the Tailwind configuration is not modified without consensus
- **AND** utility classes follow the existing naming and organization patterns

#### Scenario: Environment variable usage
- **GIVEN** the application requires configuration values

- **AND** runtime configuration avoids hardcoding values
