## Purpose

The Routes domain defines the application's page structure and navigation. It implements the user-facing views for browsing events, viewing event details, displaying calendars, and accessing various informational pages.

## Requirements

### Requirement: Home Page Event Listing
The system SHALL provide a home page that displays events with filtering and search capabilities.

#### Scenario: Display events on home page
- **GIVEN** events are loaded in the event store
- **WHEN** the home page (+page.svelte) is rendered
- **THEN** events are displayed in a list format
- **AND** each event shows title, date, time, location, and cost
- **AND** events can be displayed in list or map view

#### Scenario: Filter events on home page
- **GIVEN** events are loaded and filters are applied
- **WHEN** the home page is rendered
- **THEN** only events matching the active filters (type, music, date) are displayed
- **AND** filter chips show active state and counts

#### Scenario: Search events on home page
- **GIVEN** events are loaded and a search query is entered
- **WHEN** the home page is rendered
- **THEN** only events matching the search query in title, venue, city, description, DJ, or organizer are displayed
- **AND** search result count is shown when filtering reduces results

#### Scenario: Toggle between list and map view
- **GIVEN** the home page is displaying events
- **WHEN** the user toggles the map view button
- **THEN** the view switches between list and map displays
- **AND** map view requires location consent and shows events grouped by geographic location

### Requirement: Event Detail Page
The system SHALL provide a detailed view for individual events.

#### Scenario: Display event details
- **GIVEN** a valid event ID
- **WHEN** the event detail page (event/[id]/+page.svelte) is accessed
- **THEN** the event's full details are displayed
- **AND** information includes title, description, date/time, location, cost, and organizer
- **AND** a map shows the event's location if geographic data is available

#### Scenario: Handle invalid event ID
- **GIVEN** an invalid or non-existent event ID
- **WHEN** the event detail page is accessed
- **THEN** an error is shown or the user is redirected appropriately

### Requirement: Calendar View
The system SHALL provide a calendar view for browsing events by date.

#### Scenario: Display calendar view
- **GIGEN** events are loaded for a month
- **WHEN** the calendar page (/calendar) is accessed
- **THEN** a monthly calendar is displayed with events marked on their dates
- **AND** users can navigate between months
- **AND** clicking a date shows events for that day

### Requirement: Informational Pages
The system SHALL provide access to informational and legal pages.

#### Scenario: Access legal pages

- **THEN** the corresponding static content is displayed
- **AND** pages are accessible via the footer or menu

#### Scenario: Access community pages
- **GIVEN** the application is running
- **WHEN** users navigate to community pages (djs, veranstalter, tanzraeume, links, newsletter)
