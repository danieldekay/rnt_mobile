## Purpose

The Stores domain manages application state using Svelte 5 runes. It provides reactive data storage for events, user preferences, filters, and other application state that needs to be shared across components and persisted during the application lifecycle.

## Requirements

### Requirement: Event State Management
The system SHALL provide a reactive store for managing event data and UI state.

#### Scenario: Store initializes with default state
- **GIVEN** the application starts
- **WHEN** the event store is created
- **THEN** it contains empty events array, loading state false, no error, empty search query, and default filters
- **AND** default filters include empty types array, null music, and "week" date filter

#### Scenario: Store loads events from API
- **GIVEN** the event store is initialized
- **WHEN** `loadEvents` is called
- **THEN** loading state is set to true
- **AND** events are fetched from the API via `fetchAllEvents`
- **AND** on success, events are stored in `allEvents` and filtered events in `events`
- **AND** loading state is set to false and error is cleared
- **AND** on error, loading state is false and error message is set

#### Scenario: Store prevents redundant requests
- **GIGEN** events have been recently loaded for the same date filter
- **WHEN** `loadEvents` is called without force refresh
- **THEN** the request is skipped and existing data is re-filtered
- **AND** no new API request is made

#### Scenario: Store handles calendar month loading
- **GIVEN** a specific month date is provided
- **WHEN** `loadCalendarMonth` is called
- **THEN** events are fetched for the full month date range
- **AND** loading and error states are managed similarly to `loadEvents`
- **AND** monthly requests are cached separately from regular date filter requests

#### Scenario: Store applies filters to events
- **GIVEN** events are loaded in the store
- **WHEN** filters are changed via `setFilters`, `toggleType`, `toggleMusic`, or `setDateFilter`
- **THEN** the events displayed (`events` property) are updated to match the filters
- **AND** filtering includes event type, music type, and text search across multiple fields
- **AND** filtering is performed client-side without additional API requests

#### Scenario: Store handles text search
- **GIVEN** events are loaded in the store
- **WHEN** search query is updated via `setSearchQuery` or cleared via `clearSearch`
- **THEN** the events displayed are filtered to match the search query
- **AND** search looks across title, venue, city, description, DJ name, and organizer name
- **AND** search is case-insensitive and matches partial text

#### Scenario: Store tracks last fetched state for optimization
- **GIVEN** events have been loaded
- **WHEN** subsequent requests are made
- **THEN** the store tracks the last fetched date filter and month key
- **AND** uses this to prevent redundant API calls when data is still fresh
- **AND** resets tracking when switching between regular and calendar month loading

### Requirement: Filter Logic Implementation
The system SHALL implement correct filtering logic for event types, music types, and search.

#### Scenario: Event type filtering works correctly
- **GIVEN** events with various type categories
- **WHEN** event type filters are toggled
- **THEN** only events with matching category slugs are included in results
- **AND** when no type filters are selected, all event types are shown
- **AND** filtering uses the `EVENT_TYPE_SLUGS` mapping from constants

#### Scenario: Music type filtering works correctly
- **GIVEN** events with various music categories
- **WHEN** music type filters are toggled
- **THEN** only events with matching music category slugs are included in results
- **AND** when no music filter is selected, all music types are shown
- **AND** filtering uses the `MUSIC_SLUGS` mapping from constants

#### Scenario: Combined filters work correctly
- **GIVEN** events with various types and music
- **WHEN** multiple filters are applied (type, music, search)
- **THEN** only events matching ALL active filters are shown
- **AND** the filtering logic correctly combines all filter conditions with AND logic
